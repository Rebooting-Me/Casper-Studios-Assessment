# Recipe Enhancement Pipeline: What I Found, What I Fixed, and What I Did Not

## How I started

I sat down with the brief and the repo and made an early decision to read the code before running anything. The brief warned that the codebase was largely autogenerated and asked the central question of whether the pipeline actually works beyond the few examples it had been demonstrated on. That kind of question rarely gets answered by a smoke test. If the system runs end-to-end and produces an output file, you have learned almost nothing about whether the output is correct. So I spent the first hour or so just reading.

The pipeline is small. Around 600 lines, four modules, one data model, one scraper. Easy to hold in your head. What was harder was reconciling the code with the product description sitting next to it. The product is described as automatically improving recipes by analyzing and applying the highest voted community-tested modifications. I read the code looking for those words.

"Highest voted" turned out to be `random.choice`. The brief capitalizes "Featured Tweaks" as a product noun. The scraper extracts them into a `featured_tweaks` array. The pipeline reads from `reviews` instead. "Modifications" (plural) maps to a `modifications_applied` list hardcoded to length one. "Analyzing" is a single LLM call against a randomly picked review, with no aggregation, no ranking, no consensus across reviews. "Applying" is a fuzzy match followed by a literal `str.replace` that can silently produce no change while the pipeline still logs it as one made.

I wanted to be careful about how to weight those findings. Some are small one-line fixes, some are structural, some are conceptual. The brief gave two cues for where to look first. One was the multi-tweak case ("I added an egg and halved the sugar"), which the code clearly cannot handle because the modification schema has a scalar `modification_type` field and the prompt asks for one category per review. The other was the question of whether the implementation scales beyond the five demo recipes. Both pointed at the same place in the code.

## Deciding how to evaluate before fixing

Before changing anything, I wanted a way to attribute failures to specific stages. If extraction is wrong, the rest of the pipeline cannot recover. If application is wrong, even perfect extraction lands in the wrong place. If the orchestration is wrong (which reviews to read, how to rank them, how to combine them), the right answer to the LLM problem still does not produce the right recipe. I sketched three layers:

- **Extraction**, in isolation. Given one review and one recipe, did the LLM enumerate the modifications, label them correctly, and reference real recipe text?
- **Application**. Given a known modification and a recipe, did the deterministic Python in stage 2 produce the intended edit?
- **End-to-end**. Given a full reviews file, did the enhanced output capture the modifications a real user would expect, and were they drawn from the highest-quality reviews?

I built the first one. I want to be honest about why and what that cost me.

Extraction is the cheapest layer to instrument. The unit of test (one review, one recipe) is small. The ground truth (which edits should appear) is concrete. I could hand-label 50 fixtures in an hour. The other two layers needed harder calls. Application is not difficult but is fiddly. End-to-end requires deciding, for each recipe, what the right aggregated answer is given a stack of reviews. That is itself a product judgment I did not feel I could make confidently in the time available.

So I started with extraction. That meant the harness I built could see the multi-tweak problem in fine detail and could not see the structural problems at all. I knew this was a path-dependent choice when I made it. It became more important later.

## The golden set and what it caught

I labeled 50 reviews across 8 recipes, balancing modification categories and including 10 that explicitly bundle multiple distinct tweaks. Each review is anchored to a concrete recipe so the groundedness check is a pure substring lookup against real ingredient and instruction lines. No LLM-as-judge at scoring time for any of the four metrics. I wanted the numbers to be reproducible.

The harness uses `gpt-4o-mini` as the model under test. The baseline came in as:

| Metric | Score |
|---|---:|
| Extraction success | 100% (50/50) |
| Edit-count parity | 76% (38/50) |
| Type accuracy | 94% (47/50) |
| Groundedness per edit | 100% (54/54) |

Two things surprised me, in opposite directions.

The first surprise was groundedness. I had ranked the risk of the LLM paraphrasing or making up recipe text as a top-three concern during the code review. The data said it was a non-issue at this scale. The model reused recipe lines verbatim in 100% of cases. That was a useful lesson about not over-indexing on what code-review intuition says before any measurement has happened.

The second surprise was the concentration of failures. Out of 14 reviews that failed at least one check, 9 of them failed for exactly the same reason: a multi-modification review collapsing to a single extracted edit. The schema and the prompt were both nudging the model to pick one theme per review and stop. The remaining failures split between two smaller patterns. One was the model inventing edits the reviewer never asked for, with the cleanest example being a review that said only "Roasted at 425 instead of 375" producing both a temperature change and a compensating cook-time reduction. The other was occasional category misclassification when the review phrasing was ambiguous.

The baseline gave me a clear primary target. It also told me my code-review priorities had been partially wrong, which is the right reason to measure before fixing.

## What I chose to fix and the part I want to be straight about

At this point the two strongest candidates for first work were these.

One was the structural defects. Making the pipeline read `featured_tweaks`, ranking reviews by rating, looping over more than one review per recipe. Each of these is a small code change. None of them needs the harness I had just built.

The other was the multi-tweak collapse. Larger change, requires the prompt and schema to be rewired, and needs the harness to confirm the LLM behavior actually changed.

I went with the multi-tweak fix. My reasoning, which I want to put in writing rather than dress up:

The structural defects are real and they violate the product description on their face. I did not need an evaluation to know they were wrong. What I would have learned by fixing them is that the pipeline now reads a different field and sorts a list differently. That is true but not interesting from an AI engineering perspective. The multi-tweak problem was the only failure mode where I could demonstrate the full loop: a labeled dataset, a measured baseline, a targeted intervention, and a re-measurement against the same dataset. That loop is the kind of work the role description asked about.

There was also a sequencing argument. If I fixed Featured Tweaks first and the pipeline started reading the right reviews tomorrow, the extractor would still collapse multi-modification reviews into one edit. A better-chosen review with most of its content silently dropped is not actually progress. So multi-tweak first, structural fixes next.

I want to be plain that this is not the only defensible choice. A reasonable engineer could argue that the structural fixes are cheaper and address the more visible product gap, and they would be right about both points. The path I took weights the AI-quality measurement work above the code-level fixes because that is what the role description asked about. Someone optimizing purely for product delivery would choose differently.

## The fix itself

The fix was smaller than the investigation that preceded it. The data model was already plural-ready. `RecipeModifier.apply_modifications_batch` was fully implemented and never called from anywhere. `EnhancedRecipe.modifications_applied` was already a list. The single-modification cap was held in place by three narrow seams: the prompt asked for one, the extractor returned one, and the enhanced generator wrapped the result in a one-element list. Around 85 lines of code across four files. No schema migration, no new methods. The most satisfying part of the change was activating code that was already written.

Re-running the harness:

| Metric | Baseline | After fix | Change |
|---|---:|---:|---:|
| Extraction success | 100% | 100% | 0 |
| Edit-count parity | 76% | 90% | +14 |
| Type accuracy | 94% | 96% | +2 |
| Groundedness per edit | 100% | 97.22% | -2.78 |

The collapsed-multi-tweak bucket went from 9 reviews to 0. The two cases I had flagged in advance as residual risks (one review where the model had previously crammed two intents into a single malformed replacement string, another with same-clause multi-intent) both recovered cleanly. The compromise payload pattern from baseline, where the model produced "0 cup white sugar, 2 cups packed brown sugar" as a single replace value, disappeared once the model had room to express each intent separately.

The regressions were predicted. Groundedness dropped slightly because giving the model more output budget let it occasionally drift on the anchor text. In one case it paraphrased "3 large apples" as "2 large apples" while extracting the correct intent. In another it produced a sequence-dependent anchor referencing text it had introduced in a previous edit. Neither is fabrication. Both are correctable with a small validation gate at the stage boundary, which I deliberately held out of this change so the parity gain could be attributed to the schema change alone.

The over-reach pattern also grew by one case. Same reason. More output budget without a corresponding anchoring instruction in the prompt. The fix is a single sentence in the prompt, held out for the same attribution reason.

## What I want to acknowledge

The most important tradeoff in this work is what the harness measures and what it does not measure. The extraction harness sees one review at a time. It cannot see the random review selection. It cannot see the ignored Featured Tweaks. It cannot see the absence of multi-review aggregation. The pipeline could be picking the wrong reviews entirely and the harness would report the same numbers.

By building only the extraction layer, I biased my own ranking of failures toward what that layer can detect. When I say multi-tweak collapse was the dominant failure in the baseline, that statement is true inside the measurement I chose. The claim that it is the largest failure in the system overall is not supported by the evidence I have. The right way to confirm or disconfirm that claim is the end-to-end layer I did not build.

A few other things worth stating plainly:

The fix corrects one specific failure shape and intentionally leaves several others in place. The structural items I listed during code review are still in the code.

Using `gpt-4o-mini` both as the model under test and as the eventual judge in a Layer 3 evaluation has a known risk. The mitigation in the design is multi-vote consensus, and for production I would use a different model family in the judge slot.

The production pipeline output is still non-deterministic because the review selector is still `random.choice`. The harness reaches the extractor directly, so the eval numbers are reproducible across runs even though the production pipeline output is not.

## What I would do next

Ordered by how much they would move the product, not by how easy they are to build:

1. **Anchor the prompt to literal review claims.** A single added line in the extraction prompt. Four of the seven remaining failures share the consistency-cascade pattern (remove an ingredient, also remove the step that uses it). Cheapest remaining correctness gain.
2. **Read Featured Tweaks and rank by votes.** Two small changes that together implement the product description literally. Smaller than the paragraph describing them.
3. **Aggregate modifications across multiple reviews per recipe.** The premise of community-tested modifications is consensus, which is invisible in single-review processing.
4. **Build the end-to-end evaluation layer.** Score whether the enhanced recipe reflects the modifications described by the top voted reviews. This is the missing measurement that would tell us whether items 2 and 3 actually deliver the product description.
5. **Validate edits at the stage boundary.** A check that each `find` string still exists in the recipe state at the moment it is applied. Addresses both the groundedness regression seen here and the latent silent-no-op pattern in the application stage that the baseline did not surface but the code makes likely.

The total scope above is roughly two days of focused work. What I delivered covers the part where evaluation was the load-bearing piece, and the harness is set up to measure the rest as it lands.

## What I am taking away from this

A few small things I want on the record.

Code review told me groundedness was going to be a top concern. The data said it was not. I want to remember that as evidence that prioritizing by code-read intuition before any measurement is genuinely unreliable, even when the intuition feels strong.

I also want to remember the methodology bias of this report. Choosing to measure one layer first changed the shape of the conclusions I can draw from the data. It does not invalidate them but it scopes them. A future version of this work should build the end-to-end layer earlier so the structural and orchestration failures are visible in the same scorecard as the LLM behavior. That is the lesson I would carry into the next pipeline like this one.

The most pleasant surprise was finding that the largest correctness improvement in the system came from three small seams and an already-written batch method that nobody had called. There is something to be said for reading the code before deciding it needs to be rewritten.

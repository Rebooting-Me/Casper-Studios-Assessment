# Goal

Take over a partially-built recipe-enhancement pipeline (autogenerated by a junior engineer), determine whether it actually works beyond the 5 supplied examples, and figure out what must be fixed. The product premise: take an AllRecipes recipe + its community "Featured Tweaks" (highest-voted, modification-bearing reviews), apply those community modifications back into the recipe, and surface line-level diffs with full attribution to the source review.

The brief's two explicit cues set the priorities:

1. Does the system parse out **all** modifications in a review? ("I added an egg and halved the sugar" = two discrete tweaks.)
2. Does the system scale beyond the 5 supplied recipes, or are there assumptions baked into the implementation that won't survive contact with new data?

Time budget: ~4 hours. Deliverables: source, written document, 5-7 min video, and this trajectory file.

## Phase 1 — Architecture Comprehension
**Objective:** Read the brief and the entire codebase end-to-end before forming any hypothesis about what's broken. Understand responsibilities, assumptions, failure modes, and — critically — where AI-generated outputs flow into downstream systems without human review.

**AI Assistance:**
- Parallel reads of all pipeline modules, prompts, test harness, scraper, one raw recipe sample, and one enhanced output sample.
- Synthesized a module-by-module breakdown (responsibility / assumptions / failure points / AI-output propagation) without reading the same file twice.

**Decisions I made during this phase:**
- **Read first, run nothing yet.** Tempting to immediately `uv run python test_pipeline.py single` and see what falls over, but the brief is explicit that the codebase is largely autogenerated and the failures are likely conceptual, not crash-y. A code-level read surfaces silent failures (e.g. no-op string replacements) that a smoke test would miss.
- **Inspect the committed enhanced JSON alongside the code.** Catching drift between what the code emits and what's on disk is faster than rerunning.

**Outcome — what the architecture actually looks like:**

The pipeline is a 3-stage batch:

```
scraper_v2.py ──► data/recipe_*.json
                       │
            ┌─ TweakExtractor  (LLM, Step 1) ──► ModificationObject
LLMAnalysisPipeline ─► RecipeModifier (fuzzy text, Step 2) ──► modified Recipe + ChangeRecord[]
            └─ EnhancedRecipeGenerator     (Step 3) ──► enhanced_*.json
```

**Findings worth tracking into Phase 2 hypothesis work:**

1. **Single random review per recipe.** `tweak_extractor.py:136` does `random.choice(modification_reviews)`. There is no "highest voted" logic anywhere — the brief's product framing ("Featured Tweaks" = highest voted) is not implemented. Two runs on the same recipe produce different outputs.

2. **Schema forces "one review → one modification category."** `ModificationObject.modification_type` is a scalar `Literal[...]`. The LLM is structurally pushed to collapse multi-tweak reviews ("halved sugar AND added egg AND baked at 375") into a single category. This is precisely the brief's first cue. The prompt reinforces this by asking for a single `modification_type` and `reasoning` per call.

3. **`build_few_shot_prompt` is dead code with a latent crash bug.** `prompts.py:53` has an unbalanced `}` in `EXTRACTION_PROMPT`. `.format()` on it would raise. The runtime path uses `build_simple_prompt` (f-string), so the bug is hidden. The 4 few-shot examples are written but never used.

4. **Silent no-op replacements.** `recipe_modifier.py` does fuzzy match (`difflib.SequenceMatcher`, threshold 0.6) to pick a candidate line, then runs literal `str.replace(edit.find, edit.replace)` on it. If `edit.find` isn't a literal substring of the matched line, the line is unchanged — but a `ChangeRecord` is still appended in some branches. "Changes made" count is unreliable. 0.6 is permissive enough that `"1 cup white sugar"` vs `"1 cup brown sugar"` (~0.83 similarity) can edit the wrong line.

5. **`validate_modification_safety` is dead code.** Fully implemented in `recipe_modifier.py:221+`, never called.

6. **Model/code drift on schema.** Committed `enhanced_10813_*.json` contains `confidence_score` fields on `ModificationApplied` and `EnhancementSummary`. Current Pydantic models don't define those fields. The on-disk artifact was generated by a different code revision. Reproducing it will silently lose data.

7. **README/code drift on model.** README says `gpt-4o-mini`; code uses `gpt-3.5-turbo` (`tweak_extractor.py:24`).

8. **CWD-coupled paths.** `pipeline.py` uses `data/enhanced` relative to cwd; `test_pipeline.py` uses `../data`. Two `data/enhanced/` directories exist (`Casper-Studios-Repo/data/enhanced/` and `Casper-Studios-Repo/src/data/enhanced/`) — outputs go to different places depending on where you launch from.

9. **Scraper heuristic gates the LLM input.** `scraper_v2.py` decides `has_modification` via a handful of broad regexes (`(more|less|extra) ([\w\s]+)`, etc.). False positives feed the LLM reviews that don't actually describe a modification; false negatives starve the pipeline of valid reviews. The LLM stage is downstream of an unaudited classifier.

10. **No evals.** "Success" in `test_pipeline.py` means "an enhanced object was produced," not "the enhancement is correct." There is no ground truth, no diff inspection, no completeness check, no regression test.

11. **AI-output propagation, ranked by load-bearing impact:**
    - LLM's `find` text → fuzzy-matched recipe line → `str.replace` mutation of the canonical recipe (no grounding check).
    - LLM's `reasoning` → concatenated into user-visible `expected_impact`.
    - LLM's `modification_type` → aggregate `change_types` distribution in the summary report.
    - Scraper regex `has_modification` → eligible pool → `random.choice` → final user-visible output (non-deterministic).

**Outcome:** Architecture and failure surface are mapped. Ready to form prioritized hypotheses about what's actually broken vs. what's merely ugly.

## Phase 1b — Product-Goal Alignment Check
**Objective:** Test the code against the brief's literal product statement: _"automatically improves recipes by analyzing and applying the highest voted community-tested modifications."_

**AI Assistance:** Mapped each qualifier in the goal sentence to the corresponding code path.

**Outcome — mismatches at every load-bearing qualifier:**

| Goal phrase | Code reality |
|---|---|
| "highest voted" | `random.choice` — no use of rating/votes anywhere. |
| "Featured Tweaks" | Scraper populates `featured_tweaks`; pipeline reads `reviews` instead and ignores the featured array entirely. |
| "community-tested" (consensus) | One review per recipe, in isolation. No cross-review aggregation. |
| "modifications" (plural) | `modifications_applied` hardcoded to a one-element list; scalar `modification_type` schema caps even single-review extraction at one category. |
| "analyzing" | Extraction only; no ranking, clustering, dedup, or scoring across reviews. |
| "applying" | Reported as applied even when `str.replace` is a silent no-op. |
| "improves" | No quality check. LLM `reasoning` is passed through verbatim as `expected_impact`. |

**Shape of the gap:** code is `pick_one_random_review → extract_one_category → maybe_apply_one_edit → write_json`. Product wants `rank_featured_tweaks → analyze_consensus → apply_all_winning_modifications → verify_improvement`. The mismatch is structural, not cosmetic — it shows up in selection, schema, application, and verification layers all at once.

## Phase 1c — Top 10 Failure Modes (prioritized)
**Objective:** Enumerate and rank the failure modes I'd expect to fire on real data, weighted by user impact × frequency × implementation risk.

**Ranking (1 = worst):**

1. **Multi-tweak reviews collapsed to one modification** — schema + prompt structural cap. Brief's #1 cue.
2. **Silent no-op `str.replace`** — fuzzy match → literal replace fails silently, but pipeline reports success. Worst-class failure: invisible.
3. **Wrong-line edit from 0.6 fuzzy threshold** — corrupts the recipe (e.g. white-sugar edit lands on brown-sugar line).
4. **Only one review processed per recipe** — `modifications_applied` hardcoded to length 1.
5. **`featured_tweaks` array ignored** — scraper extracts it, pipeline never reads it.
6. **`random.choice` selection** — ignores votes/ratings/featured flag; non-deterministic output.
7. **No grounding of LLM `find` text against recipe before Stage 2** — root cause of #2 and #3. `validate_modification_safety` is implemented and uncalled.
8. **Scraper `has_modification` regex false positives → LLM fabricates edits** for reviews that don't actually describe a modification.
9. **Edit schema (replace/add/remove only) can't represent splits, reorders, unit changes, global temp changes** — forces miscategorization.
10. **LLM `reasoning` surfaced verbatim as user-facing `expected_impact`** — ungrounded text reaches the product surface.

**Key insights from the ranking:**
- #1, #2, #4 fail *invisibly*: the pipeline reports success. The eval harness in Phase 2 has to detect "claimed enhanced but wasn't."
- #2, #3, #7 are one chain (ungrounding → fuzzy match → literal replace). A single grounding gate between Stage 1 and Stage 2 dissolves all three.
- #1, #4, #5, #6 are all "one shot when the product wants many" — the architecture's most pervasive bias.

# Validation

## Phase 2 — Evaluation Framework Design
**Objective:** Design a three-layer eval before touching code. Constraints: prefer objective metrics; use gpt-4o-mini as the SUT and as a judge only where NL is unavoidable.

**Design principles:**
- One layer per stage so failures localize.
- Hand-labeled golden set over the 6 real recipes + synthetic stress set + a "negatives" set (regex-flagged reviews that contain no actual modification, to measure hallucination on bad upstream input).
- LLM-as-judge restricted to one metric (faithfulness), with strict rubric, temp 0, n=3 majority vote.
- No subjective metrics ("does this taste better?") — out of scope.

**Layer 1 — Extraction (deterministic):** edit recall, edit precision, edit-count delta (the headline for multi-tweak collapse), type accuracy, groundedness, hallucination rate on negatives.

**Layer 2 — Application (no LLM):** intent-realization, no-op rate (directly measures the silent-replace bug), wrong-line rate, idempotency, phantom-change rate. Inputs are hand-written ModificationObjects to keep Stage 1 noise out.

**Layer 3 — End-to-end:**
- Objective: coverage vs golden, diff↔citation symmetry, citation integrity (source text must verbatim-appear in input), structural integrity, determinism (fixed seed + temp 0), selection-quality proxy (avg rating of selected vs avg rating of eligible reviews — quantifies "highest voted").
- LLM-judged: faithfulness — per change, does it reflect what the reviewer said? gpt-4o-mini, n=3 majority, JSON output.

**Cost:** ~70 gpt-4o-mini calls per full eval pass — cheap enough for per-change runs.

**Mapping to Phase 1c failure modes:** every top-10 failure has at least one metric in the framework that would catch it. Headline metrics:
- #1 collapsed multi-tweak → `edit_count_delta`, `edit_recall`
- #2 silent no-op → `no_op_rate`, `diff_citation_symmetry`
- #3 wrong-line edit → `wrong_line_rate`, `faithfulness_score`
- #6 random selection → `selection_quality`, `determinism`
- #7 ungrounded `find` → `groundedness`
- #8 fabricated edits → `hallucination_rate` on negatives

**Decisions I made:**
- Hand-label rather than synthesize the entire golden set. The brief's correctness is about *real* review language, not idealized prompts.
- Keep Layer 2 LLM-free. The application stage is deterministic Python; testing it with deterministic inputs removes LLM variance from the signal.
- Restrict the judge to one bounded NL question ("does change X reflect statement Y"). Refused to use it for taste/quality.

## Phase 2b — Golden Dataset (50 reviews)
**Objective:** Produce the Layer 1 / Layer 3 ground truth: 50 hand-labeled (review → expected edits) fixtures covering every modification category and explicitly including multi-modification reviews.

**AI Assistance:** Drafted 8 recipe templates (anchored to the 6 supplied recipes plus 2 plausible additions for generalization headroom), wrote 50 review texts in casual AllRecipes voice, and labeled each with structured edits. Every `find` string is a literal substring of its anchored recipe so the deterministic groundedness check runs without external lookups.

**Output:** [evals/golden_reviews.json](Casper-Studios-Repo/evals/golden_reviews.json)

**Distribution:** 8 addition + 8 removal + 8 substitution + 8 quantity + 4 time + 4 temp + 10 multi-mod. Multi-mod block contains 28 expected edits across 10 reviews (2.8 edits/review average; max 4).

**Decisions I made:**
- **Anchor every edit to literal recipe text.** Makes the groundedness check pure substring; no LLM needed at eval time for Layer 1's grounding metric.
- **Express same-line multi-intent as separate atomic edits** (e.g. GR-046's "doubled cheese AND used gruyere" → two edits sharing one `find`), with a `matcher_notes` clause that says the eval is satisfied if the final line reflects all intents regardless of whether the extractor emits 1 or 2 edits. Keeps ground truth honest about user intent while not over-prescribing extractor output shape.
- **No edge-case negatives in this file.** Phase 2 framework defines a separate negatives set for hallucination measurement. Mixing them here would muddy precision/recall scoring.
- **Real-recipe language.** Reviews include filler praise ("Big hit with the kids", "tastes like nonna made it") so extraction is tested against natural language, not lab speech.

## Phase 3 — Baseline Extraction Measurement
**Objective:** Wire the eval harness to the existing `TweakExtractor` (no production code changes), run gpt-4o-mini against all 50 golden reviews, report metrics and group failures by root cause.

**Harness:** [evals/run_extraction_eval.py](Casper-Studios-Repo/evals/run_extraction_eval.py) — pure measurement; imports `TweakExtractor` via `sys.path` insertion; passes `model="gpt-4o-mini"` at constructor.

Decision I made: asked the user before installing rather than fabricating numbers — a baseline that isn't real has zero value.

**Run cost:** 50 sequential API calls, ~80 seconds wall time.

### Headline results

| Metric | Score |
|---|---:|
| Extraction success rate | **100%** |
| Edit-count parity | **76%** |
| Modification-type accuracy | **94%** |
| Groundedness per edit | **100%** |

### Failures grouped by root cause

- **R1 — Single-`modification_type` schema forces multi-tweak collapse** — 9 reviews (90% of all multi-mod reviews). GR-050 is the most diagnostic case: model packed multi-intent into one malformed `replace` payload (`"0 cup white sugar, 2 cups packed brown sugar"`), which Stage 2 would faithfully execute, producing a broken recipe.
- **R2 — Domain over-reach: LLM fabricates cascading edits the reviewer didn't request** — 3 reviews. **Not predicted in Phase 1c.** Example: GR-040 review said only "Roasted at 425 instead of 375," LLM also reduced cook time to compensate. GR-016 removed garlic per user but also removed the cooking step that uses garlic.
- **R3 — Modification-type taxonomy maps poorly to nuanced reviews** — 3 reviews. Mostly downstream of R1+R2 — when forced to a single scalar `modification_type`, the LLM picks substitution-flavored language for ambiguous reviews (e.g. "Skipped the mustard" → classified as substitution because "Just as good without" reads as a sub for nothing).

### Phase 1c predictions vs reality

**Confirmed:** Phase 1c #1 (multi-tweak collapse) is the dominant failure, at the exact rate the schema would predict.

**Disconfirmed at this dataset scale:** Phase 1c #7 (ungrounded `find` strings) — 0/54 predicted edits failed groundedness. gpt-4o-mini reuses recipe text verbatim. Caveat: clean machine-formatted recipes here; real AllRecipes data has more noise. Also Stage 2 can still fail despite grounded `find` (see GR-050).

**New finding not in Phase 1c:** Domain over-reach (R2). Implication for fixes: a naive "extract all modifications" prompt could amplify R2. The fix needs to enumerate multiple modifications AND anchor strictly to literal review claims.

### Revised fix priority (data-informed)

1. Schema + prompt redesign for multi-modification per review — single biggest correctness lever (9 of 13 failures).
2. Stage 2 sanity gate on `replace`/`add` payloads — not on `find` grounding (it's already 100%) but on payload validity.
3. Anti-overreach instruction: extract only literal review claims, no consistency inference.

**Outputs:** [evals/results/baseline_extraction_report.json](Casper-Studios-Repo/evals/results/baseline_extraction_report.json), [evals/results/baseline_failure_report.md](Casper-Studios-Repo/evals/results/baseline_failure_report.md).

**Decisions I made:**
- Override the repo's default `gpt-3.5-turbo` to `gpt-4o-mini` per Phase 2 design — measuring the model the assessment actually targets.
- Do not modify production code in this phase — baseline must be of *the current system*, not an in-progress fix.
- Synthesize the diagnostic script as throwaway (deleted post-run) rather than committing it — eval harness stays focused.

## Phase 4 — Re-ranked Failure Modes + Solution Candidates for R1
**Objective:** Reconcile Phase 1c priorities with Phase 3 empirical results, then design (not implement) three candidate fixes for R1.

**Re-ranked priority (top 6):**

1. **R1 — Multi-tweak collapse.** Confirmed dominant. 90% of multi-mod reviews fail.
2. **#5 — `featured_tweaks` ignored.** Structural product gap; trivial to fix.
3. **#4 — One review per recipe.** Hardcoded `modifications_applied = [one]`; trivial after R1.
4. **#6 — `random.choice` selection.** No vote/rating ranking anywhere.
5. **#2 — Silent no-op `str.replace`.** Layer 2 not yet measured, but R1's GR-050 case (LLM packed multi-intent into one malformed replace) makes this *logically certain* to fire when R1 is fixed naively.
6. **R2 — Domain over-reach.** New failure mode discovered in baseline; not in Phase 1c. 6% empirical rate. Cheap to mitigate via prompt anchoring.

**Demoted:** Phase 1c #7 (ungrounded `find`) — empirically 0% at baseline scale.

**Three solution candidates for R1:**

- **A. Schema expansion to `List[ModificationObject]`** — single LLM call returns a list. Activates the existing dead `apply_modifications_batch` code path. Medium effort, medium risk. ~96% parity ceiling. Risk: may amplify R2.
- **B. Two-pass enumerate-then-structure** — Stage 1a lists atomic claims, Stage 1b structures each one. Highest completeness ceiling AND best R2 mitigation (per-claim anchoring). Highest cost (N+1 LLM calls per review) and complexity. Per-claim attribution is a free bonus for the line-level diff UI.
- **C. Heuristic clause-splitter wrapping the existing extractor** — split review on conjunctions/lists, one LLM call per clause. Lowest effort, no schema changes. Residual failure floor: same-clause multi-intent (GR-046 "doubled cheese AND used gruyere" — one clause, two intents). Doesn't help R2.

**Decision framing (not yet decided):** A is the highest leverage at the assessment's 4-hour budget — it directly removes the schema cap and unlocks already-written batch code. B is product-aligned (per-claim citations match the line-level diff UI) but expensive. C is the emergency-ship option with a known residual failure floor.

**Cross-cutting observation:** A naive R1 fix (just lift the schema cap) will likely *increase* the Stage 2 silent-no-op rate, because the LLM's compromise behavior on multi-intent (e.g. GR-050's `"0 cup white sugar, 2 cups packed brown sugar"`) becomes more frequent. Any R1 fix should be paired with a Stage 2 sanity gate on `replace`/`add` payloads.

## Phase 5 — Option A Deep Trace + MVP Sizing
**Objective:** Map every code site that assumes single-modification cardinality, size the change, and define the smallest viable Option A.

**Key discovery:** the data model and one critical stage are *already plural-ready*. The single-modification cap is held in place by only 3 narrow seams (prompt template, extractor return type, enhanced_generator's L138 hardcoded `[mod]` wrap). `RecipeModifier.apply_modifications_batch` is fully implemented dead code; `EnhancedRecipe.modifications_applied` is already `List`. This is the most leverage per LOC I've seen in the codebase — wiring up what's already written.

**MVP scope:** ~50 LOC across 4 files (prompts.py, tweak_extractor.py, pipeline.py, enhanced_recipe_generator.py). No schema change, no new methods, no migration. ~30 min focused implementation + ~10 min eval harness adapter.

**Expected improvement (grounded in per-review failure analysis):**
- Edit-count parity: 76% → realistic 90% (range 84-94% depending on same-line multi-intent recovery)
- Modification-type accuracy: ~unchanged (94% → 93-96%)
- Groundedness: unchanged (~100%)
- Stage 2 silent-no-op risk: **reduced**, because the LLM no longer needs to encode multi-intent into one compromise payload (the GR-050 pattern disappears)

**What MVP does NOT fix:**
- R2 over-reach (separate prompt-anchoring line, intentionally out of MVP scope to isolate the R1 lever)
- #4 (one review per recipe — only the per-review cap is lifted)
- #5/#6 (`featured_tweaks` / random selection)
- Same-clause / same-line multi-intent (GR-046, GR-050) — residual floor; possibly addressed by Option B but not A

**Decision framing (not yet decided):** MVP is sized to fit easily inside the assessment budget and produces an immediately measurable delta against the baseline. The eval harness from Phase 3 reruns with ~10 LOC of changes.

**Decisions I made:**
- Hold R2 mitigation out of MVP. Isolating one variable per measurement makes attribution clean. R2 prompt line is a separate cheap follow-up.
- Don't migrate `ModificationObject` itself — keep the model class scalar-typed, just allow many of them. This avoids ripples into `recipe_modifier.py` and preserves the existing dead-code path that already does what we want.
- Don't preserve old method signatures for back-compat — the call sites are inside the package, one caller each. Migration overhead outweighs the cost of a clean rewrite.

## Phase 6 — Option A MVP Implementation + Validation
**Objective:** Land the smallest viable change to lift the single-modification cap. Re-run eval, measure delta in isolation.

### Rationale
The baseline confirmed R1 (multi-tweak collapse) as the dominant correctness failure (90% of multi-mod reviews, 18% overall). Phase 5 trace showed the data model and `apply_modifications_batch` were already plural-ready — the cap was held in place by exactly 3 narrow seams. Highest leverage-per-LOC opportunity in the codebase.

### Implementation summary
Four files changed; ~85 LOC delta. No new methods, no schema migration, no test additions.

- **[prompts.py](Casper-Studios-Repo/src/llm_pipeline/prompts.py)**: `build_simple_prompt` rewritten to ask for `{"modifications": [...]}`. Explicit instruction that "reviews often describe MULTIPLE distinct modifications" with an example. Intentionally did NOT add R2 anti-overreach line to isolate the R1 lever.
- **[tweak_extractor.py](Casper-Studios-Repo/src/llm_pipeline/tweak_extractor.py)**: `extract_modification` returns `List[ModificationObject]` instead of `Optional[ModificationObject]`. Empty list signals both "no modifications" and extraction failure (same semantics as the old `None` return). Bumped `max_tokens` 1000 → 1500 to fit longer multi-modification responses. `extract_single_modification` tuple shape updated.
- **[pipeline.py](Casper-Studios-Repo/src/llm_pipeline/pipeline.py)**: swapped `apply_modification` → existing `apply_modifications_batch` (formerly dead code). Iterates over returned `List[List[ChangeRecord]]`. Builds `List[ModificationApplied]` with all entries pointing to the same `source_review`.
- **[enhanced_recipe_generator.py](Casper-Studios-Repo/src/llm_pipeline/enhanced_recipe_generator.py)**: `generate_enhanced_recipe` accepts `modifications: List[ModificationObject]` and `change_records_per_modification: List[List[ChangeRecord]]`. Removed the L138 hardcoded `[modification_applied]` wrap; uses a list comprehension paired with `zip()`.
- **[evals/run_extraction_eval.py](Casper-Studios-Repo/evals/run_extraction_eval.py)**: harness adapter — consumes the list, flattens edits for parity scoring, treats type accuracy as "any predicted type intersects expected types." Added `--label` CLI flag so baseline file stays intact.

`models.py`, `recipe_modifier.py`, `__init__.py`, `test_pipeline.py` — all unchanged.

### Validation results

| Metric | Baseline | Option A MVP | Δ |
|---|---:|---:|---:|
| Extraction success | 100% | 100% | 0 |
| **Edit-count parity** | **76%** | **90%** | **+14 pp** |
| Modification-type accuracy | 94% | 96% | +2 pp |
| Groundedness per edit | 100% | 97.22% | −2.78 pp |
| Fully-grounded reviews | 100% | 96% | −4 pp |

**Review status changes (50 total):** 9 recovered, 2 regressed, 2 shifted, 3 unchanged (out of MVP scope).

- **R1 dissolved.** `collapsed_to_one_edit` bucket: 9 → 0. All multi-mod cases now extract the expected number of modifications. Even the Phase 4-predicted residual cases (GR-046 same-clause multi-intent, GR-050 same-line replace conflict) recovered cleanly.
- **R2 modestly amplified.** Over-extraction bucket grew 3 → 4 (added GR-014: removed thyme AND removed the instruction step that mentions thyme). Predicted in Phase 5 as a risk of giving the LLM more output rope without anti-overreach anchoring.
- **Groundedness regression**, 2 cases:
  - GR-007: LLM paraphrased `"3 large apples"` as `"2 large apples"` in the `find` anchor. Quantity drift.
  - GR-040: LLM emitted a sequence-dependent anchor — referenced text it had introduced in a prior edit (`"Preheat oven to 425 degrees F."`). Even a substring-against-original-recipe gate wouldn't catch this; it requires multi-edit sequence awareness.
- **GR-050 compromise payload risk dissolved**, as Phase 4 predicted: now three clean modifications instead of one malformed `replace` cramming multi-intent into a single line.

### Remaining limitations (intentionally not in MVP)

- **R2 over-reach** (GR-011, GR-014, GR-016, GR-040). Next obvious fix is the anti-overreach prompt line; one-line change.
- **R3 type taxonomy misfit** (GR-013). Cosmetic; downstream of phrasing ambiguity.
- **#4** — pipeline still picks one review per recipe. R1 fix is per-review uncapped; per-recipe still N=1.
- **#5** — `featured_tweaks` array still ignored. The product premise (highest-voted modifications) still not implemented.
- **#6** — `random.choice` over the eligible review pool, no vote/rating ranking.
- **#2** — Stage 2 silent no-op replace not yet measured. Layer 2 eval not built. With Option A reducing the GR-050-style compromise payloads, the likely fire rate is *lower* than the baseline would have predicted, but a Stage 2 eval is still needed to verify.

### Cross-cutting note
The two groundedness regressions point at a small new failure shape: with more output budget, the LLM occasionally drifts on the *anchor* even when getting the intent right. GR-007 is a number swap; GR-040 is a sequence dependency. Neither is a structural failure; both are correctable with a per-edit grounding gate, but that's a separate (small) change.

**Decisions I made:**
- Did NOT add the R2 anti-overreach prompt line, even though I knew it was a one-line cheap win, because the user asked to isolate R1's impact. Now have clean attribution: the +14pp parity gain is entirely the schema-cap fix.
- Increased `max_tokens` from 1000 to 1500. Multi-modification responses are longer; under the old budget GR-047 (4 expected edits) could plausibly truncate. Minor, but worth noting as a change beyond pure structural rewiring.

## Phase 7 — Post-Fix Failure Triage
**Objective:** Classify the 7 remaining post-MVP failures by root cause and bucket, and explain MVP's intentional deferrals.

**Distribution across the user's three buckets:**

| Bucket | Reviews | Count |
|---|---|---:|
| R2 over-reach | GR-011, GR-014, GR-016, GR-040 | 4 |
| Groundedness drift | GR-007, GR-040 (compound) | 2 |
| Structural #4/#5/#6 | — | **0** |
| Outside the three buckets | GR-013 (R3 type misfit), GR-048 (residual same-line compression) | 2 |

**Key observations:**

- **R2 is now the dominant remaining failure** (4/7). All four are "consistency cascade" patterns — remove ingredient → also remove the step that uses it; change temp → also change time. Fixable with one prompt-level anti-overreach line. Deferred from MVP to keep R1's attribution clean.
- **Groundedness regressed 2 cases.** GR-007 is straightforward (LLM paraphrased a quantity in the anchor). GR-040 is more interesting: a *sequence-dependent* anchor — text the LLM introduced in a prior edit. Even a substring-against-original-recipe gate would not catch it; needs multi-edit sequence-aware validation. New failure shape, not in Phase 1c.
- **#4/#5/#6 are invisible to the Layer 1 harness.** They live at the pipeline-orchestration level (one-review-per-recipe, `featured_tweaks` ignored, `random.choice`), not at the per-review extraction level. Confirming and measuring them needs a Layer 3 end-to-end eval against multi-review per-recipe ground truth — separate phase.
- **GR-048 is a near-miss**, not a structural failure: 3 of 4 expected edits extracted; the LLM compressed temp+time into one instruction edit. Down from 90% of multi-mods (baseline) to 1 isolated case (post-MVP). Driving this residual floor to zero needs Option B or a richer edit schema — larger surface than MVP allows.

**Why MVP intentionally deferred each category:**

- **R2:** isolating one variable per measurement. Adding the anti-overreach line in the same change would have confounded the +14pp parity attribution.
- **Groundedness drift:** the right fix is a Stage-1→Stage-2 grounding gate, which is architectural, not a prompt tweak. Outside cardinality scope.
- **#4/#5/#6:** require Layer 3 measurement infrastructure and bigger product-level shifts. Each warrants its own design phase.
- **R3 type misfit:** cosmetic in current pipeline (type flows into summary stats but not into recipe content).
- **Same-line compression:** schema expansion got it from 9 cases to 1; further reduction needs different tools (Option B, structural schema).

**Net read:** the remaining failure surface (7/50 = 14% across any check; 86% all-checks-pass) is small, well-localized, and dominated by R2. Neither R2 nor the groundedness regressions are structural — both are addressable with prompt-level changes. The brief's product-premise gaps (#4/#5/#6) remain the largest *unmeasured* problem area; they're not in this report because the harness doesn't see them.
